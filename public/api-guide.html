Odoo API Developer Guide

คู่มือเชื่อมต่อดึงข้อมูลระบบ Odoo 14 ผ่าน REST API สำหรับนักพัฒนาและทีมที่ได้รับอนุญาต
📄 เวอร์ชัน: กรกฎาคม 2026 (API v2.0.0) 🌐 Base URL: http://192.168.101.104/Odoo 🔒 ใช้ได้เฉพาะเครือข่ายภายในบริษัท (LAN)

1ภาพรวม

แอปของคุณ (ใน LAN) ──HTTP/JSON──>  http://192.168.101.104/Odoo/api/odoo/*
                                        │  API Gateway (ควบคุมสิทธิ์การอ่าน/เขียน)
                                        ▼
                                  ฐานข้อมูล Odoo 14 ของบริษัท
  • Base URL: http://192.168.101.104/Odoo — เรียกได้เฉพาะจากเครือข่ายภายในบริษัทเท่านั้น
  • API เป็น read-only เป็นหลัก — การเขียนข้อมูล (INSERT) ต้องใช้ token และเปิดสิทธิ์รายตาราง
  • ข้อมูลอ่านจากฐานข้อมูลจริงของ Odoo → ได้ข้อมูลชุดเดียวกับที่เห็นใน Odoo
  • ทุก response เป็น JSON รูปแบบเดียวกัน:
// สำเร็จ
{ "success": true, "data": ..., "count": 123 }

// ผิดพลาด
{ "success": false, "error": "คำอธิบายข้อผิดพลาด" }

การขอสิทธิ์ใช้งาน

ต้องการต้องทำอะไร
อ่านข้อมูล (Read API)ไม่ต้องขอ token — เครื่องที่อยู่ใน LAN บริษัทเรียกได้เลย
เขียนข้อมูล (INSERT)ติดต่อผู้ดูแลระบบ (ฝ่าย IT/EDP) เพื่อ (1) รับ token และ (2) ขอเปิดสิทธิ์ตารางที่ต้องการเขียน

2เริ่มต้นเร็วที่สุด (Quick Start)

# 1) เช็คว่า server ออนไลน์
curl http://192.168.101.104/Odoo/health
# → {"status":"ok","message":"Odoo dashboard server is running"}

# 2) ดูรายชื่อตารางทั้งหมดใน Odoo
curl http://192.168.101.104/Odoo/api/odoo/tables

# 3) ยิง SQL query แรกของคุณ
curl -X POST http://192.168.101.104/Odoo/api/odoo/query \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT id, name FROM res_partner ORDER BY id LIMIT 5"}'

3Read API (ไม่ต้องใช้ token)

GET/health — Health check

ตรวจว่าระบบทำงานอยู่ ใช้สำหรับ monitoring ได้

GET/api/odoo/tables — รายชื่อตารางทั้งหมด

คืนรายชื่อตารางทั้งหมดในฐานข้อมูล Odoo — ใช้สำรวจว่ามีข้อมูลอะไรให้ดึงบ้าง

{ "success": true, "data": [ { "table_name": "account_move" }, ... ], "count": 500 }

GET/api/odoo/schema/:tableName — โครงสร้างตาราง

ดูคอลัมน์ ชนิดข้อมูล และ nullable ของตารางที่สนใจ

curl http://192.168.101.104/Odoo/api/odoo/schema/res_partner

# → { "success": true, "data": [
#      { "column_name": "id", "data_type": "integer", "is_nullable": "NO" }, ... ] }

POST/api/odoo/query — SQL query อ่านอย่างเดียว ⭐

Endpoint หลักสำหรับดึงข้อมูล — ส่ง SQL ของคุณเองได้อิสระ (อ่านอย่างเดียว)

  • อนุญาตเฉพาะคำสั่งที่ขึ้นต้นด้วย SELECT หรือ WITH คำสั่งเดียว (ห้ามมี ; คั่นหลายคำสั่ง) — คำสั่งอื่นถูกปฏิเสธด้วย HTTP 403
  • ใส่ LIMIT เสมอ — query ยิงเข้าฐานข้อมูลที่ระบบงานจริงใช้อยู่ (ดูข้อ Fair Use)
  • Timestamp ในฐานข้อมูลเก็บเป็น UTC — แปลงเป็นเวลาไทยด้วย AT TIME ZONE 'UTC' AT TIME ZONE 'Asia/Bangkok' (คอลัมน์ชนิด DATE เช่น invoice_date ใช้ตรง ๆ ได้)
curl -X POST http://192.168.101.104/Odoo/api/odoo/query \
  -H "Content-Type: application/json" \
  -d '{"sql": "SELECT name, email FROM res_partner WHERE customer_rank > 0 ORDER BY id DESC LIMIT 20"}'

# → { "success": true, "data": [ { "name": "…", "email": "…" } ], "count": 20 }

4Write API — INSERT (ต้องใช้ token)

⚠️ คำเตือนสำคัญ: การ INSERT ผ่าน API นี้เขียนตรงเข้าฐานข้อมูล จะข้าม business logic ของ Odoo ทั้งหมด (เลขรันเอกสาร, ฟิลด์คำนวณ, การตัดสต๊อก, workflow) — ระบบจึงเปิดให้เขียนเฉพาะตารางที่ผู้ดูแลอนุญาตเป็นรายตารางเท่านั้น ตารางเอกสารหลักของ Odoo (sale_order, account_move, stock_move ฯลฯ) จะไม่ได้รับอนุญาตในทุกกรณี

การยืนยันตัวตน

ทุกคำขอเขียนต้องแนบ token ที่ได้รับจากผู้ดูแลระบบ อย่างใดอย่างหนึ่ง:

Authorization: Bearer <token>     ← แนะนำ
x-api-key: <token>                ← ทางเลือก

POST/api/odoo/insert/:table — เพิ่มข้อมูล

  • Body: { "data": {…} } (แถวเดียว) หรือ { "data": [{…}, …] } (หลายแถว สูงสุด 500)
  • หลายแถวรันใน transaction เดียว — แถวไหนพัง rollback ทั้งหมด
  • ชื่อคอลัมน์ถูกตรวจกับ schema จริงก่อนเสมอ — คอลัมน์ที่ไม่มีในตารางจะได้ 400 พร้อมระบุคอลัมน์ที่ผิด
  • เติม ?dryRun=1 เพื่อดู SQL ที่จะรันโดยไม่เขียนจริง — ใช้ทดสอบก่อนยิงจริงทุกครั้ง
# ทดสอบก่อน (ไม่เขียนจริง)
curl -X POST "http://192.168.101.104/Odoo/api/odoo/insert/my_table?dryRun=1" \
  -H "Authorization: Bearer <token>" -H "Content-Type: application/json" \
  -d '{"data": {"name": "ทดสอบ", "qty": 5}}'

# ยิงจริง
curl -X POST http://192.168.101.104/Odoo/api/odoo/insert/my_table \
  -H "Authorization: Bearer <token>" -H "Content-Type: application/json" \
  -d '{"data": [{"name": "A", "qty": 1}, {"name": "B", "qty": 2}]}'

Response สำเร็จ — คืนแถวที่ insert แล้วทั้งหมด (รวมค่า default เช่น id):

{ "success": true, "count": 2, "data": [ { "id": 101, "name": "A", "qty": 1 }, … ] }

5ข้อควรรู้เมื่อดึงข้อมูล Odoo

ตารางหลักที่ใช้บ่อย

ตารางเก็บอะไร
res_partnerลูกค้า / ผู้ขาย / ผู้ติดต่อ
product_product + product_templateสินค้า (join ผ่าน product_tmpl_id)
sale_order / sale_order_lineใบสั่งขาย
account_move / account_move_lineใบแจ้งหนี้ ใบลดหนี้ และรายการบัญชี
stock_quantสต๊อกคงเหลือรายตำแหน่ง (กรอง location_id ที่ stock_location.usage='internal' = ของในคลังจริง)
pos_orderบิลขายหน้าร้าน (POS)

นับยอดขายให้ตรงกับฝ่ายบัญชี

ถ้าต้องดึง "ยอดขาย" ให้ใช้เงื่อนไขชุดนี้ ไม่เช่นนั้นตัวเลขจะไม่ตรงกับรายงานของบริษัท:

  • นับเฉพาะ account_move ที่ move_type IN ('out_invoice','out_refund') และ state = 'posted' (Draft ไม่นับ)
  • จำนวนเงินใช้ -aml.balance จาก account_move_line = ยอดก่อน VAT ในสกุลเงินบริษัท (invoice เป็นบวก, ใบลดหนี้เป็นลบ)
  • กรองเฉพาะบรรทัดสินค้า: aml.exclude_from_invoice_tab = false AND aml.display_type IS NULL
  • "วันนี้" ของธุรกิจ = (now() AT TIME ZONE 'Asia/Bangkok')::date เทียบกับ invoice_date
-- ตัวอย่าง: ยอดขายสุทธิเดือนนี้
SELECT SUM(-aml.balance) AS net_sales
FROM account_move_line aml
JOIN account_move am ON am.id = aml.move_id
WHERE am.move_type IN ('out_invoice','out_refund')
  AND am.state = 'posted'
  AND aml.exclude_from_invoice_tab = false
  AND aml.display_type IS NULL
  AND date_trunc('month', am.invoice_date) =
      date_trunc('month', (now() AT TIME ZONE 'Asia/Bangkok')::date)

6Error Codes

HTTPความหมายเจอเมื่อ
400คำขอผิดรูปแบบไม่ส่ง sql, คอลัมน์ไม่มีในตาราง, data เกิน 500 แถว, ข้อมูลขัด constraint ของตาราง
401token ผิดหรือไม่ได้ส่งWrite API ทุกตัว
403ไม่อนุญาตquery ที่ไม่ใช่ SELECT/WITH, ตารางยังไม่ได้รับสิทธิ์ insert
404ไม่พบตารางไม่มีในฐานข้อมูล
503Write API ยังไม่เปิดใช้งานระบบยังไม่เปิดรับการเขียนข้อมูล — ติดต่อผู้ดูแล
500ข้อผิดพลาดฝั่งระบบดูรายละเอียดในฟิลด์ error แล้วแจ้งผู้ดูแล

ทุก error มี body: { "success": false, "error": "…" }

7ตัวอย่างโค้ดภาษาต่าง ๆ

JavaScript (fetch)

const BASE = 'http://192.168.101.104/Odoo';

const q = await fetch(`${BASE}/api/odoo/query`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ sql: 'SELECT name FROM res_partner LIMIT 5' }),
}).then(r => r.json());

if (!q.success) throw new Error(q.error);
console.log(q.data);

Python (requests)

import requests

BASE = 'http://192.168.101.104/Odoo'

# SQL query
r = requests.post(f'{BASE}/api/odoo/query',
                  json={'sql': 'SELECT name FROM res_partner LIMIT 5'})
body = r.json()
assert body['success'], body.get('error')
print(body['data'])

# INSERT (ต้องมี token + ตารางได้รับสิทธิ์แล้ว)
r = requests.post(f'{BASE}/api/odoo/insert/my_table',
                  headers={'Authorization': 'Bearer <token>'},
                  json={'data': {'name': 'ทดสอบ', 'qty': 5}})

PHP (cURL)

$base = 'http://192.168.101.104/Odoo';

$ch = curl_init("$base/api/odoo/query");
curl_setopt_array($ch, [
    CURLOPT_RETURNTRANSFER => true,
    CURLOPT_POST => true,
    CURLOPT_HTTPHEADER => ['Content-Type: application/json'],
    CURLOPT_POSTFIELDS => json_encode(['sql' => 'SELECT name FROM res_partner LIMIT 5']),
]);
$body = json_decode(curl_exec($ch), true);
curl_close($ch);
if (!$body['success']) { die($body['error']); }
print_r($body['data']);

8ข้อควรปฏิบัติ (Fair Use)

  1. ใส่ LIMIT ทุก query — ฐานข้อมูลนี้คือตัวเดียวกับที่ระบบงานหลักของบริษัทใช้อยู่ query หนักกระทบผู้ใช้ Odoo ทั้งบริษัท
  2. อย่า SELECT * จากตารางใหญ่ (account_move_line, stock_move_line มีหลายล้านแถว) — เลือกคอลัมน์และกรองช่วงเวลาเสมอ
  3. อย่า polling ถี่เกินจำเป็น — ข้อมูลส่วนใหญ่ไม่ได้เปลี่ยนทุกวินาที ตกลงรอบดึงข้อมูลที่เหมาะกับงานของคุณกับผู้ดูแล
  4. การเขียนข้อมูล ให้ทดสอบด้วย ?dryRun=1 ก่อนทุกครั้ง
  5. ข้อมูลที่ได้เป็นข้อมูลภายในบริษัท — ห้ามนำไปเผยแพร่หรือส่งต่อภายนอกโดยไม่ได้รับอนุญาต
  6. ถ้าต้องใช้ query ซับซ้อนซ้ำ ๆ เป็นประจำ แจ้งผู้ดูแลให้ทำเป็น endpoint เฉพาะ (มี cache) จะเร็วกว่าและเบากับระบบกว่า
